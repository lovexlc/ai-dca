# 基金持仓收益工具需求文档

## 1. Summary

在现有站点中新增一级导航 **基金持仓收益**，提供一个面向单组合持仓管理的工具页。用户可上传持仓截图，由 OCR 解析出基金代码、基金名称、买入均价、持有份数等字段；识别结果进入可编辑表格，支持手动新增、修改、删除。页面每次打开时通过 **Worker 净值接口** 拉取最新日净值与上一交易日净值，计算 **今日收益** 和 **总收益**。净值查询增加共享缓存层与 `data/` 缓存文件，便于不同用户命中相同持仓请求时直接返回。前端在净值更新过程中提供逐行“已更新”提示，并在组合级今日收益上显示统一的“更新中 / 已更新 / 部分更新失败”状态。

## 2. Goals

- 降低手工录入基金持仓的成本。
- 让用户基于最新净值快速查看组合收益。
- 保留人工修正能力，避免 OCR 误识别导致结果不可用。
- 不影响现有 `基金切换` 页面与流程。

## 3. Core Requirements

### 3.1 入口与页面形态

- 新增一级 Tab：`基金持仓收益`。
- 与现有 `基金切换` 页面并存，互不替换、互不复用业务流程。
- 页面包含 3 个主要区域：
  - 上传导入区
  - 收益摘要区
  - 持仓明细表格区

### 3.2 持仓导入

- 支持上传 `PNG / JPG / JPEG` 持仓截图。
- OCR 识别目标为 **当前持仓列表**，不是交易流水。
- 识别后生成当前组合草稿，不直接静默覆盖已有数据。
- 默认导入策略：**替换当前组合**。
- 可选交互：后续可保留“追加导入”入口，但本版本默认以替换为主。

### 3.3 持仓明细管理

- 一行表示一只基金的当前持仓。
- 用户可手动：
  - 修改任意识别结果
  - 新增一行持仓
  - 删除单行
  - 清空全部持仓
- 字段要求：
  - 基金代码：6 位数字，必填
  - 基金名称：选填
  - 买入均价：大于 0，必填
  - 持有份数：大于 0，必填
- 若 OCR 未可靠识别出 `代码 / 买入均价 / 份数`，该行不导入，并在警告中说明。

### 3.4 收益计算

- 使用 **最新公布单位净值** 与 **上一交易日单位净值** 计算收益。
- 计算公式固定：
  - `买入成本 = 买入均价 × 持有份数`
  - `当前市值 = 最新净值 × 持有份数`
  - `今日收益 = (最新净值 - 上一交易日净值) × 持有份数`
  - `总收益 = (最新净值 - 买入均价) × 持有份数`
- 页面展示：
  - 组合总成本
  - 组合当前市值
  - 组合今日收益
  - 组合总收益
  - 每只基金的最新净值、净值日期
- 若某只基金净值拉取失败，仅该行显示异常提示，不阻塞其余持仓计算。

### 3.5 自动更新与状态反馈

- 不做后台定时任务。
- 页面每次打开时自动请求最新净值并重算。
- 前端净值请求统一走 Worker 接口，不直接从浏览器访问上游净值源。
- 若拉取成功，更新本地缓存中的净值快照。
- 若拉取失败，保留上次成功快照并提示失败状态。
- 某一行在成功获取到 `最新净值 + 上一交易日净值` 后，该行显示 `已更新`。
- 组合级 `今日收益` 卡片在仍有任一持仓处于更新过程时，显示 `更新中`，不提前展示部分结果。
- 仅当全部持仓都完成净值获取后，组合级 `今日收益` 才显示最终值，并切换为 `已更新`。
- 若存在部分持仓更新失败，则失败行单独提示错误，组合级 `今日收益` 显示 `部分更新失败`。

### 3.6 Worker 接口与共享缓存

- OCR 与净值查询都继续通过 Worker 提供接口。
- 净值查询按“基金代码集合”作为缓存粒度，Worker 需要先对代码数组执行排序、去重，再生成稳定 cache key。
- 仓库 `data/` 目录下增加净值缓存文件，作为共享缓存基线，建议路径：
  - `data/holdings-nav-cache/index.json`
  - `data/holdings-nav-cache/<cache-key>.json`
- 当不同用户请求相同持仓代码集合时，Worker 命中共享缓存后直接返回。
- 仅当缓存不存在、缓存已过期，或请求代码集合发生变化时，Worker 才回源拉取最新净值。
- 缓存命中与否需要在接口响应中返回，便于前端展示与调试。

### 3.7 本地保存

- 仅支持 **单持仓组合**。
- 数据保存在当前浏览器本地存储中。
- 刷新页面后应恢复上次保存的持仓数据与最近一次成功净值快照。
- 不做账号体系、云端同步、跨设备同步。

## 4. UI/UX 设计规范

### 4.1 整体设计风格

- 延续现有项目的视觉体系，不另起一套全新风格。
- 整体定位为 **轻量金融工作台**。
- 风格关键词：
  - 干净
  - 稳定
  - 现代
  - 可编辑
  - 数据优先
- 设计定调：
  - 浅色系
  - 卡片化
  - 大圆角
  - 轻玻璃感
  - 强可读性、弱装饰

### 4.2 色彩规范

- 主色：`indigo / 蓝紫`
  - 用于主按钮、当前激活态、上传入口、核心操作。
- 正收益：`emerald / 绿色`
  - 用于盈利数字、上涨状态、成功提示。
- 负收益：`red / 偏玫红红色`
  - 用于亏损数字、错误状态、风险提示。
- 中性色：`slate / 灰蓝`
  - 用于背景、边框、辅助文案、未激活状态。

### 4.3 组件风格

- 卡片：
  - 使用大圆角，建议 `24px ~ 32px`。
  - 白底或半透明白底。
  - 轻阴影，不要厚重投影。
- 收益卡：
  - 数字层级高于说明文案。
  - 今日收益与总收益必须有明显正负色区分。
- 表格：
  - 不做传统硬边框表格。
  - 采用“可编辑工作表”风格，支持行内编辑。
- 按钮：
  - 主按钮高对比，突出核心动作。
  - 次按钮、辅助按钮弱化处理。
- 状态标签：
  - 使用 pill/tag 风格呈现，如 OCR 状态、净值更新时间、异常状态。
- 上传区：
  - 作为页面主视觉，使用大面积拖拽上传卡片。

### 4.4 页面布局

页面结构固定为三段：

1. 顶部上传 Hero
   - 中间大上传卡
   - 展示 OCR 进度、识别状态、错误提示
2. 中部收益摘要
   - 总成本
   - 当前市值
   - 今日收益
   - 总收益
   - 净值更新时间
3. 底部持仓明细
   - 支持新增、编辑、删除、清空
   - 每行展示净值、今日收益、总收益

### 4.5 交互规范

- 主流程采用 **先识别，再校正**。
- 用户编辑买入价、份数、代码后应即时重算，不增加额外确认成本。
- 错误提示优先采用就地提示，不弹过重的模态框。
- 收益数字需要明显正负反馈，但不使用夸张动画。
- 上传完成后应直接进入可编辑结果态，减少额外跳转。
- 行级净值更新状态与组合级收益更新状态需要分开表达，避免用户误以为组合收益已经完整刷新。
- 行内状态文案固定为：`更新中`、`已更新`、`更新失败`。
- 组合级今日收益状态文案固定为：`更新中`、`已更新`、`部分更新失败`。

### 4.6 与现有项目保持一致

- 延续现有项目中的：
  - `slate` 背景体系
  - `indigo` 主操作色
  - 大圆角卡片
  - 顶部 tab 工作台
  - 简洁数据卡 + 可编辑表格
- 视觉上保持“同一产品内新增一个工作区”的连续性，而不是新产品落地页。

### 4.7 参考其他 Tab 的现有设计语言

结合当前 `加仓计划`、`定投计划`、`交易计划中心`、`交易历史` 四个 Tab，新增持仓页应继续复用以下共性：

- 顶部导航：
  - 使用现有 sticky 顶部导航条。
  - 导航栏保持半透明浅底、毛玻璃、圆角 pill 激活态。
  - 新增 Tab 后整体观感仍应像一个统一工作台，而不是嵌入外部子页面。
- 首屏摘要：
  - 页面内容开始区域优先出现一排 `4` 张摘要卡。
  - 每张卡维持统一结构：`eyebrow / 核心数值 / note`。
  - 默认以白卡为主，只给最重要指标一到两张加 `indigo` 或 `emerald` 强调。
- 卡片体系：
  - 常规信息使用白底卡片 + 浅边框 + 轻阴影。
  - 次级说明区使用 `slate-50` 或浅色背景卡片。
  - 强调区可参考其他 Tab 使用轻渐变卡：
    - 浅色渐变：`from-indigo-50 via-white to-white`
    - 深色渐变：`from-slate-900 via-slate-800 to-indigo-900`
- 信息分组：
  - 每个模块都要有清晰的 `SectionHeading`，包含 eyebrow、title、description。
  - 不同模块之间靠卡片和留白分组，不依赖厚重分隔线。
- 状态表达：
  - 标签统一用 pill/tag 风格。
  - 成功、同步中、错误、提醒等状态优先通过 pill 和轻提示卡表达，而不是大面积告警条。
- 空状态：
  - 沿用其他 Tab 的虚线边框空状态卡。
  - 空状态里要给出一句解释 + 1 到 2 个明确操作按钮。

### 4.8 页面布局向其他 Tab 对齐

- 内容宽度统一控制在 `max-w-6xl` 级别。
- 首屏采用与其他 Tab 一致的“摘要卡在上，主体卡片在下”的工作台结构。
- 桌面端推荐布局：
  - 第一行：4 张摘要卡
  - 第二行开始：主内容区 + 辅助信息区的双栏布局
  - 主内容区优先放上传结果和持仓编辑表
  - 辅助信息区放净值更新状态、OCR 告警、接口提示、操作说明
- 页面不应做成全屏单列表；应保留类似 `定投计划` 和 `交易计划中心` 的“主编辑区 + 侧边说明区”层次。
- 若存在关键主操作（如“保存当前持仓”），可参考 `定投计划` 使用底部悬浮操作条；若实时自动保存，则不需要为了形式强行加底栏。

### 4.9 组件细节向其他 Tab 对齐

- 按钮：
  - 主按钮沿用现有 `indigo` 高对比按钮。
  - 次按钮沿用 `slate` 浅底按钮。
  - 危险操作如删除、清空使用弱化红色边框按钮，不做高饱和实心红。
- 表单：
  - 输入框保持 `slate-50` 底色，focus 后切白底与 `indigo` 边框。
  - 下拉、切换器、分段选择器的样式与现有 `SelectField`、频率按钮组一致。
- 数据卡：
  - 标题小写眉标使用大写风格和加宽字距。
  - 核心数字使用较粗字重和紧凑字距。
- 表格/列表：
  - 表头与正文使用弱分层，不做重网格线。
  - 行 hover 仅做浅色变化，避免重交互噪音。
- Toast 与反馈：
  - 继续复用现有右上角 toast 体系，操作成功/失败提示风格统一。

### 4.10 持仓页的具体落地风格

- `基金持仓收益` 页不是纯上传页，而是“工作台型收益页”：
  - 顶部保留摘要卡
  - 中部放核心编辑与收益区域
  - 右侧放更新状态和辅助说明
- 上传区可以保留较强视觉识别，但不能压过整个工作台风格。
- 收益摘要卡建议参考其他 Tab 的统计卡语言，避免做成完全不同的大屏报表。
- 可以保留 1 张较强视觉卡片作为“组合总览 / 净值状态总览”，但其余卡片仍应回到现有白卡体系。
- 组合级今日收益、总收益、当前市值等信息应和首页/定投页一样，先保证可读性，再考虑视觉强化。

### 4.11 移动端风格要求

- 移动端继续沿用当前项目的卡片堆叠式布局。
- 参考首页移动端折叠区与现有 mobile guardrails：
  - 不把标题、状态、指标、按钮全部压进一行
  - 允许 badge 和状态单独换行
  - 数字与说明分层展示
- 若持仓表在移动端过宽，应降级为卡片列表或横向滚动编辑区，但视觉语言仍需和桌面端一致。
- 移动端上传区、收益摘要区、持仓区之间要有清晰层级，不做过度紧凑排版。

## 5. Public Interfaces / Types

### 5.1 前端持仓行结构

```ts
type HoldingRow = {
  id: string;
  code: string;
  name: string;
  buyPrice: number;
  shares: number;
  buyAmount: number;
  latestNav: number;
  latestNavDate: string;
  previousNav: number;
  previousNavDate: string;
  navStatus: 'idle' | 'updating' | 'updated' | 'error';
  navFetchedAt: string;
  navMessage: string;
};
```

### 5.2 OCR 接口

`POST /api/holdings/ocr`

- 请求：`multipart/form-data`
  - `file`: 持仓截图
- 响应：

```json
{
  "rows": [
    {
      "code": "006113",
      "name": "某某基金",
      "buyPrice": 1.8326,
      "shares": 1200
    }
  ],
  "warnings": [],
  "recordCount": 1,
  "confidence": 0.92,
  "durationMs": 1800
}
```

### 5.3 净值接口

`POST /api/holdings/nav`

- 请求：

```json
{
  "codes": ["006113", "159941"]
}
```

- 响应：

```json
{
  "cacheKey": "006113_159941",
  "cacheHit": true,
  "allUpdated": true,
  "items": [
    {
      "code": "006113",
      "latestNav": 1.8326,
      "latestNavDate": "2026-04-07",
      "previousNav": 1.8404,
      "previousNavDate": "2026-04-03",
      "status": "updated",
      "message": "已更新",
      "source": "shared-cache"
    }
  ],
  "errors": []
}
```

### 5.4 共享缓存文件

- `data/holdings-nav-cache/index.json`
  - 维护已生成的缓存 key、对应代码集合、最近更新时间。
- `data/holdings-nav-cache/<cache-key>.json`
  - 保存单次代码集合请求的净值快照结果。
- 缓存文件内容与 `POST /api/holdings/nav` 的响应主体保持一致，便于 Worker 直接回传。

## 6. Implementation Constraints

- 继续复用现有 React 工作区结构。
- 继续复用现有 Cloudflare Worker 体系。
- 前端不直接访问净值上游，统一通过 Worker。
- 不修改现有 `/api/ocr` 的基金切换识别逻辑。
- 新持仓功能使用独立接口与独立 OCR Prompt。
- `data/` 目录中需要加入净值共享缓存文件，并随前端发布产物一起同步到 `docs/data/`。
- 净值来源固定为东财历史净值接口解析最近两条记录。
- 支持普通场外基金与可返回净值历史的 ETF 代码。

## 7. Acceptance Criteria

### 7.1 功能验收

- 上传持仓截图后，能识别出至少部分有效持仓行。
- OCR 结果可手动编辑并即时重算收益。
- 支持手动新增、删除单行、清空全部。
- 页面刷新后数据仍可恢复。
- 页面打开时会自动刷新净值并更新收益。
- 同一持仓代码集合的重复请求可由 Worker 共享缓存直接返回。
- 行级可以看到 `已更新` 状态，组合级今日收益在全部完成前持续显示 `更新中`。
- 净值缺失时页面不崩溃，能明确提示异常代码。

### 7.2 测试场景

- 单只基金持仓导入成功并计算收益。
- 多只基金持仓导入成功并汇总组合收益。
- OCR 误识别后手动修正买入价与份数。
- 删除一行后组合汇总正确变化。
- 净值接口部分失败时，其余基金仍正常计算。
- 相同代码集合重复请求时，Worker 返回共享缓存命中结果。
- 某一行先完成更新时，该行先显示 `已更新`，但组合级今日收益仍显示 `更新中`。
- 全部行完成后，组合级今日收益才切换为 `已更新`。
- 场外基金代码与 ETF 代码各验证一组。
- 构建后新 Tab 在 `docs/` 发布产物中可正常访问。

## 8. Non-goals

- 不支持多组合管理。
- 不支持按历史买入批次回放成本。
- 不支持盘中估值优先。
- 不支持后台每日抓快照。
- 不支持收益提醒或推送通知。

## 9. Assumptions

- “买入价格”按 **当前持仓成本均价** 处理。
- 用户上传的是券商/基金 App 的持仓列表截图，而非成交记录截图。
- 今日收益采用 **最新日净值 - 上一交易日净值** 计算。
- 若截图中名称与代码同时存在，优先以代码作为净值查询主键。
- 默认只维护一份当前组合；如后续需要多组合，再独立扩展数据模型。
- 共享缓存按“排序去重后的代码集合”作为唯一键，不按用户维度拆分。
